Why Hash When You Can Array?

A Question of Convenience

April 25, 2015

If you're anything like me, your first thought upon seeing a hash was, "But this is just an array of ordered pairs..." It wasn't clear to me why one would need a separate variable type to do what we could already do with arrays. But I was missing the point. I don't know how ruby compilers are written, but I imagine that behind the scenes hash objects are simply arrays of ordered pairs with predefined methods to save us all time. Because the smart people who created ruby knew we would all want to implement the same functionality on our own if they didn't do it for us.

So really the question is not "why a hash as opposed to an array," but "why a hash at all." And the answer ‐ like many answers in computer science ‐ boils down to strategic laziness. Specifically, hashes afford the following advantages:

  1. Hashes save the programmer the trouble of remembering details. I may not be able to memorize 500 full names, but a hash can.
  2. Hashes reduces code repetition. Rather than typing 312-555-8754, I can type phone["sam"]. This may not seem like much of a reduction, but (1) the compiler auto-filled the variable names, and (2) my next program may require 487-bit strings, not phone numbers.
  3. Hashes promote distribution of labor. The developer using the hash doesn't need to know any of the individual elements ofthe hash, only the basic parameters of what it contains.

So, having made these points, when should one use hashes? Consider the following example: Two developers are co-designing a video game. One might develop graphics and layouts while the other develops the engine and objects to populate the game world. When the former decides that level 7 should contain two orcs and an elf for the player to fight, she doesn't want to worry about the details of creating orcs or elves. She just wants to ask for two orcs and an elf. Now, Developer 2 could create two object classes "orc" and "elf" to define these characters, but Developer 1 may find it much easier if Developer 2 creats a "character" class, which contains "orc" and "elf" as sub-classes. Developer 1 can then specify the character's race, which is the key to a hash that populates the details of the relevant sub-type. Developer 1 writes less code and gets to ignore the details, while Developer 2 is free to change the details of the various sub-types without forcing Developer 1 to re-write any code. Everyone wins!.
